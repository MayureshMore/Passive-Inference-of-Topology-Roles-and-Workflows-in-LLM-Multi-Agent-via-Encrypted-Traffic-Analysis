# RECREATE — build this entire project from scratch

This is the full rebuild bible: how to reproduce **every dataset, model, and result**
starting from an empty machine. For a one-command demo see [START_HERE.md](START_HERE.md);
for the frozen-feature reproduction path see [DATA.md](DATA.md); for the numbers themselves
see [RESULTS.md](RESULTS.md).

---

## 0. Two levels of "from scratch" — pick one

| Level | What you rebuild | Inputs you need | Determinism |
|---|---|---|---|
| **A. Re-analyse** | all tables/figures from the **published feature matrices** | `a2a-features.zip` (see [DATA.md](DATA.md)) + this repo | **Exact** — RF/GBT/transfer are seed-fixed |
| **B. Re-collect** | the **whole dataset**, then all results | this repo + the testbed + LLMs below (no archive) | **Statistical** — fresh captures land inside the published CIs, not on the exact numbers |

Level A is minutes on a laptop, no GPU/network/LLM. Level B is the true "from scratch" and
is what the rest of this document covers. Raw collection is stochastic by nature (live LLM
generation over a real network), so Level B reproduces the **findings**, not the identical
digits — this is expected and documented in [DATA.md](DATA.md).

---

## 1. What you must supply (not in the repo)

The code is fully here; these external inputs are not, and you provide them:

| Input | Why | Where it's used |
|---|---|---|
| **Ollama** + models `llama3.2:3b` and `qwen2.5:7b` | local LLM inference per site (A uses llama, B uses qwen) | all agent deployments |
| **A capture host** you own (macOS or Linux) with `tcpdump`/BPF | the passive tap | every capture step |
| **(C5 only) a 2nd host + VPN** — the paper used a Dell in India over WireGuard | real WAN vantage | §6, [C5 runbook](docs/C5_WAN_RUNBOOK.md) |
| **(Off-the-shelf only) Google `a2a_mcp` sample** + a **Gemini API key** | external, independently-authored A2A system | §7, [PHASE5 patches](docs/PHASE5_A2A_MCP_PATCHES.md) |
| **WAN host IPs** to fill into `configs/testbed_wan.yaml` (`FILL_IN_*`) | the config ships with placeholders | §6 |

Everything else (agents, feature pipeline, models, evaluators, configs) is committed.

## 3. Environment

```bash
git clone <repo> && cd <repo>
python3 -m venv venv && source venv/bin/activate
pip install -r requirements.txt

# LLMs (per capture host)
#   install Ollama: https://ollama.com
ollama pull llama3.2:3b     # deployment A / A-model-swap control
ollama pull qwen2.5:7b      # deployment B / logic-swap control
```

**Pins that matter:** `a2a-sdk==0.3.26` (the 1.x line dropped the Starlette HTTP/SSE app and
is gRPC-only — unusable for this traffic study) and `langgraph==1.2.6` (deployment C). Python
3.11+.

## 4. Sanity gate — prove A2A traffic is on the wire (do this first)

```bash
python scripts/run_poc.py --mode local --use-llm --model llama3.2:3b
```

Expect `OVERALL : PASS`. This must pass before any collection — it confirms the agents speak
JSON-RPC/SSE and that the capture actually sees the flows.

## 5. Full rebuild in one command (Deployments A/B + swaps + defenses + background)

`scripts/rebuild_all.sh` **is** the from-scratch driver for the LAN experiments. It resets
the working dirs, collects all four deployments + two live-defended sets + real background,
extracts features for each, and re-runs the whole LAN evaluation suite:

```bash
bash scripts/rebuild_all.sh 50 256      # 50 traces per (workflow×topology) pair, 256 output-token cap
```

What it produces (data dir → experiment), all writing into fresh user-owned dirs:

| Deployment / set | Collector invocation | Data dir |
|---|---|---|
| **A** (A-logic + llama3.2:3b) — headline | `run_pilot.py --model llama3.2:3b` | `data/processed` |
| **A-model** (A-logic + qwen2.5:7b) | `run_pilot.py --model qwen2.5:7b` | `data/processed_amodel_sdk` |
| **B** (B-logic + qwen2.5:7b) | `run_pilot.py --deployment b --model qwen2.5:7b` | `data/processed_b_sdk` |
| **B-logic** (B-logic + llama3.2:3b) | `run_pilot.py --deployment b --model llama3.2:3b` | `data/processed_blogic_sdk` |
| **rate** defense (live) | `run_pilot.py --defense rate` | `data/processed_defense_rate` |
| **pad** defense (live) | `run_pilot.py --defense pad` | `data/processed_defense_pad` |
| **background** (real hard negatives) | `collect_background.py` | `data/processed_background_sdk` |

Then it runs: closed-world (`evaluate.py`), cross-deployment A↔B, the model-vs-logic 2×2,
open-world-vs-background, and the live C4 defense overhead+degradation. Results land in
`data/results/`.

> **Capture privileges:** collection needs BPF access. On the paper's Mac testbed, BPF perms
> allowed capture without `sudo`; otherwise prefix with `sudo`, or grant once with
> `chmod o+r /dev/bpf*` (macOS) / `setcap cap_net_raw+ep $(command -v tcpdump)` (Linux).
> Snaplen is fixed at 96 bytes and the BPF filter is scoped to the agent ports — **header
> metadata only, never payload**.

### Deployment C (LangGraph runtime-invariance control) — separate step

C is not in `rebuild_all.sh` (it's the cross-framework control). Collect it with the same
pilot driver against the LangGraph orchestrator, then run the control + diagnostic:

```bash
# collect C (reuses A's specialists; LangGraph StateGraph orchestrator, ports 8030–8033)
python scripts/run_pilot.py --deployment langgraph --model llama3.2:3b --out data/raw_langgraph --num-predict 256
python scripts/extract_features.py --raw data/raw_langgraph --out data/processed_langgraph --scapy

# A↔C control (relabels the B-slot as C; suppresses the false "generalizes" verdict)
python scripts/evaluate_cross_deployment.py --dir-a data/processed \
    --dir-b data/processed_langgraph --label-b C --control \
    --out data/results/cross_framework.json
python scripts/diagnose_runtime_traffic.py --dir-a data/processed --dir-c data/processed_langgraph \
    --out data/results/runtime_traffic_diagnostic.json
```

## 6. Cross-network C5 (US ⇄ India WAN) — full procedure

This needs a second host and a VPN; the complete step-by-step (both hosts, capture vantage,
gates, troubleshooting) is in **[docs/C5_WAN_RUNBOOK.md](docs/C5_WAN_RUNBOOK.md)**. In short:

```bash
# remote (India):  serve the specialist agents
venv/bin/python scripts/serve_agents.py --topology star --deployment a

# local (US): drive the orchestrator so A2A crosses the VPN; capture POST-decap on the tunnel iface
sudo venv/bin/python scripts/collect_wan.py \
  --remote-host <INDIA_IP> --iface <vpn-tunnel-iface> \
  --deployment a --topology star --n 50 --num-predict 256 --out data/raw_wan

venv/bin/python scripts/extract_features.py --raw data/raw_wan --out data/processed_wan --scapy
venv/bin/python scripts/evaluate_c5.py --local data/processed --wan data/processed_wan
# → data/results/c5_cross_network.json + figure
```

Fill the `FILL_IN_*` hosts in `configs/testbed_wan.yaml` first. Capture on the **tunnel
interface** (e.g. `utun8`), not the physical NIC, or you capture the encrypted outer tunnel.

## 7. Off-the-shelf external system (Google `a2a_mcp`, Phase 5)

Tests the attack on a system **you did not build**. Setup, the 8 compatibility patches, and
the multi-turn driver are in **[docs/PHASE5_A2A_MCP_PATCHES.md](docs/PHASE5_A2A_MCP_PATCHES.md)**
and [`third_party/a2a_mcp/`](third_party/a2a_mcp/). Needs a Gemini API key (paid tier ≈
$0.043/trace; the paper collected 150 traces for $6.39).

```bash
scripts/collect_offtheshelf.sh                    # drive orchestrator over A2A + capture (ports 10100–10105)
python scripts/extract_offtheshelf.py             # features (IPv4 AND IPv6 — the sample binds ::1)
python scripts/evaluate_offtheshelf_detection.py  # binary A2A-vs-background detection (AUC)
python scripts/analyze_offtheshelf_topology.py    # hub-and-spoke recovery from headers alone
```

Labels don't align with our taxonomy, so this corroborates **detection** and **topology
observability** only — no role/workflow transfer number.

## 9. Verify the rebuild

```bash
pytest -q                              # 30 tests: features, defense, metrics, stats/SSE, deep shapes
bash scripts/reproduce.sh              # re-analyse frozen features into a sandbox, diff vs committed
```

`reproduce.sh` prints a side-by-side of reproduced vs committed macro-F1 and **never touches**
`data/results/`. For a fresh full run into a dated copy:

```bash
A2A_RESULTS_DIR=data/results_$(date +%F) bash scripts/reproduce.sh --full-suite
```

If you re-collected (Level B), your point estimates should land inside the published 95% CIs
in [RESULTS.md](RESULTS.md); they will not equal the frozen digits, and that's correct.

## 10. Ethics (non-negotiable, baked into the tooling)

- **Metadata only** — 96-byte snaplen, BPF filter scoped to the agent host:port flows; no
  payload is ever written to disk (`capture/recorder.py` enforces this).
- **Owned infrastructure only** — never point the collector at systems you don't operate.
  The off-the-shelf capture drives a *local* copy of `a2a_mcp` you run yourself.
- **Defenses evaluated before release** — C4 mitigation numbers ship alongside the attack.
